## 1. 한눈에 보는 결과

### 실제 LB 제출 기록 (4번 제출)

| 제출 파일 | 모델 구성 | OOF | LB | 등수 | 평가 |
|---|---|---|---|---|---|
| `AH_03_신정호_v3.csv` | LightGBM 단일 (94 features) | 0.74045 | 0.74201 | 3등 | ✅ 검증된 베이스 |
| `8조_팔로리안_v6.csv` | LGBM (123 features, 가이드 +29) | 0.74039 | 0.74188 | - | ❌ 떨어짐 |
| `8조_팔로리안_v8.csv` | LGBM + Target Encoding (97 features) | 0.74047 | 0.74210 | - | ⚪ 미미 |
| **`8조_팔로리안_v9.csv`** | **6-way 앙상블 (LGBM + XGB + CatBoost)** | **0.74100** | **0.74221** | **2등** | ⭐ **현재** |

### LB 점수 진화

```
v3: 0.74201 (3등)
 │
 ├─ v6: 0.74188  ❌ 가이드 +29 추가하니 떨어짐 (-0.00013)
 ├─ v8: 0.74210  ⚪ Target Encoding으로 미세 개선 (+0.00009)
 └─ v9: 0.74221  ⭐ 6-way + CatBoost로 의미 있는 개선 (+0.00019, 2등 달성, 최종 제출)
```

### OOF-LB 전이율 분석 — 의사결정의 핵심 신호

| 변화 | OOF Δ | LB Δ | 평가 |
|---|---|---|---|
| v3 → v6 | -0.00006 | -0.00013 | 실패 (가이드 피처 노이즈) |
| v3 → v8 | +0.00002 | +0.00009 | 미미 (TE는 트리에 거의 무효) |
| v3 → v9 | **+0.00055** | **+0.00019** | **성공** (CatBoost로 진짜 다양성) ⭐ |

**핵심 발견**: OOF가 살짝 올라가도 LB가 안 따라옴. **다양성이 진짜 추가될 때만 LB가 의미 있게 움직임**. v9에서 CatBoost를 추가하니 비로소 LB가 +0.00019 만큼 의미 있게 상승.

## 2. 개선 여정 — 의사결정의 흐름

```

   │
   │  Q1: 어떻게 점수를 올릴까?
   │  → 도메인 가이드의 29개 피처 추가가 가장 직관적
   │
   ▼
[v6 시도 + 제출] 가이드 피처 +29개 추가 (123 features)
   │  결과: LB 0.74188 (-0.00013) ❌
   │  학습: "트리 모델에는 도메인 피처 추가가 무관 (가이드 Part 07도 명시)"
   │
   ▼
[방향 전환] 피처가 안 되면 → 모델 다양성으로
   │  Q2: XGBoost를 추가하면 다른 신호 잡을까?
   │  → 시도했으나 LGBM↔XGBoost 상관 0.9964 (같은 패밀리)
   │  → 진짜 다양성 부족 판단, LB 미제출
   │
   ▼
[v8 시도 + 제출] Target Encoding 3개 추가 (97 features)
   │  결과: OOF 0.74047, LB 0.74210 (+0.00009) ⚪
   │  학습: "LB 살짝 올랐지만 v6 마이너스 회복 정도"
   │
   ▼
[결정적 분기] 진짜 다른 알고리즘 = CatBoost
   │  Q3: oblivious tree + ordered TS는 LGBM/XGBoost와 다른 실수를 할까?
   │  → 학습 결과: 단독 OOF 0.74045 (v3와 동일)
   │  → 핵심: 다른 모델과 상관 0.9916~0.9932 (이전 0.99~0.998 대비 확연히 낮음)
   │  → "같은 점수여도 다른 행을 다르게 예측" → 진짜 다양성
   │
   ▼
[v9 = 6-way 앙상블 제출] 모든 자산 결합 ⭐ 최종 제출
   │  구성: LGBM v3, LGBM+TE, XGB(2종), 분리 모델, CatBoost
   │  결과: OOF 0.74100 (+0.00055), LB 0.74221 (+0.00019)
   │  → 등수 3등 → 2등
```

## 3. v3 베이스라인 — 출발점으로 선택한 이유

### 왜 v3 노트북을 출발점으로 선택했나

**v3는 이미 LB 0.74201로 검증된 모델**이었습니다. 새로운 시도가 실패해도 돌아갈 안전망이 필요했고, v3는 잘 만들어진 베이스였습니다:

1. **안정적 LightGBM 단일**: 5-seed × 5-fold 배깅으로 분산 줄임
2. **검증된 도메인 피처 5개**: 가이드 29개 중 임상적으로 가장 의미 있는 5개만 선별
3. **Native categorical 처리**: pd.Categorical → LightGBM 직접 전달, 결측 대치 없음

## 4. 학습시킨 도메인 지식 — IVF 7단계와 임상 지식

가이드 (`2일차기록.html`)는 **IVF 시술의 임상 도메인 지식**을 체계적으로 정리한 문서. 이 지식을 어떻게 모델에 학습시켰는지 정리합니다.

### 4.1. IVF 시술의 7단계 — 모델이 이해해야 할 흐름

```
STEP 01: 배란 유도 (난포 자극)
   호르몬제로 여러 난포 동시 자극 (자연 1~2개 → 시술 10개+)
   [컬럼] 배란 유도 유형
   ▼
STEP 02: 난자 채취 (Egg retrieval)
   난포에서 난자를 바늘로 흡입 (수면마취)
   [컬럼] 수집된 신선 난자 수, 난자 채취 경과일
   ▼
STEP 03: 정자 준비
   남편 또는 기증자의 정자 활동성 좋은 것 선별
   [컬럼] 정자 출처
   ▼
STEP 04: 수정 (Fertilization)
   IVF (자연수정) or ICSI (정자 1개 직접 주입)
   [컬럼] 혼합된 난자 수, 미세주입된 난자 수
   ▼
STEP 05: 배양 (Embryo culture)
   3일(분할기) 또는 5~6일(배반포)간 인큐베이터
   [컬럼] 총 생성 배아 수
   ▼
STEP 06: 이식 또는 동결
   질 좋은 배아 1~3개 이식, 나머지 동결 보관
   [컬럼] 이식된 배아 수, 저장된 배아 수
   ▼
STEP 07: 착상 및 임신 확인
   2주 후 hCG 혈액검사로 임신 여부 확인
   [TARGET] 임신 성공 여부
```

이 7단계를 모델이 학습하도록 **각 단계의 컬럼들을 모두 피처로 포함**.

### 4.2. 핵심 임상 용어와 모델 적용

| 용어 | 정의 | 모델에 어떻게 반영했나 |
|---|---|---|
| **배아 (Embryo)** | 정자 + 난자가 만나 형성된 초기 생명 | `총 생성 배아 수`, `log_n_embryo` (한계효용) |
| **배반포 (Blastocyst)** | 배양 5~6일차 배아 (성공률 가장 높음) | `has_BLASTOCYST` 토큰, `days_mix_to_transfer`로 배양 기간 추출 |
| **분할기 (Cleavage)** | 배양 2~3일차 배아 | 배양 기간 차이로 구분 |
| **착상 (Implantation)** | 배아가 자궁벽에 붙는 과정 | (직접 측정 불가, target에 반영) |
| **신선 주기 (Fresh)** | 채취 → 수정 → 바로 이식 | `신선 배아 사용 여부`, `해동된 배아 수 == 0` |
| **동결 주기 (FET)** | 동결 배아를 녹여 이식 | `동결 배아 사용 여부`, `해동된 배아 수 > 0` |
| **ICSI (미세주입)** | 정자 1개를 난자에 직접 주사 (남성요인 난임) | `has_ICSI`, `미세주입된 난자 수`, `icsi_ratio` |
| **PGT-A / PGT-M** | 착상 전 유전 선별검사 | `PGS 시술 여부`, `PGD 시술 여부`, `miss_gene_test` |

### 4.3. 시술 유형 3가지 — 성공률 차이

| 시술명 | 방법 | 성공률 | 모델에 반영 |
|---|---|---|---|
| **IUI (인공수정)** | 정자를 자궁에 직접 주입 | 10~15% | (이 데이터엔 거의 없음) |
| **IVF (체외수정)** | 난자+정자를 실험실에서 만남 | 25~30% | `시술 유형 == 'IVF'`, `has_IVF` |
| **ICSI (미세주입술)** | 정자를 난자에 직접 주입 | 27~30% | `has_ICSI`, `미세주입된 난자 수` |

### 4.4. 연령별 성공률 — 가장 강한 단일 예측 변수

| 연령대 | 난자 질 | IVF 성공률 | 특징 |
|---|---|---|---|
| 20대~34세 | 최상 | **35~45%** | 표준 프로토콜, 양질 배아 |
| 35~37세 | 양호 | 30~35% | 이식 개수 증가 고려 |
| 38~40세 | 감소 | 20~25% | PGT-A 권장 시작 |
| 41~42세 | 빠른 감소 | 10~15% | 단일 시도로 어려움 |
| **43세 이상** | **심각한 감소** | **5% 이하** | **기증 난자 고려** |

**모델에 반영한 핵심 피처**:
- `age_num` (시술 당시 나이 수치): 범주 → 중위값 매핑 (만18-34 → 26, ..., 만45-50 → 47.5)
- **`real_age_num` (실제 난자 제공자 나이)** ⭐: 자가시술이면 본인 나이, **기증이면 기증자 나이** (43세 이상 → 기증 회피 패턴 포착)
- `age_x_embryo_ratio` (나이 × 이식률 상호작용): 고령일수록 이식 개수 늘림

### 4.5. 가이드의 4가지 핵심 임상 통찰

#### ① 배아 수 역U자 패턴 (총 생성 배아 수)
```
7개 미만 (67.7%): 22.07% 성공률 (고령 + 저반응)
7~20개 (31.5%):  35.03% 성공률 ← 최적
20개 초과 (0.7%): 22.64% 성공률 (PCOS + 과반응, freeze-all 전략)
```
**모델 반영**: `log_n_embryo`로 비선형 변환

#### ② ever_delivered = 가장 강력한 단일 예측 변수
> "과거에 출산했다는 건 '임신과 출산이 가능한 몸'의 증명"
> Templeton 1996, Barnett-Itzhaki 2020 등 수십 년 논문이 검증

**모델 반영 (v6)**: `ever_delivered = 총 출산 횟수 > 0`

#### ③ is_FET 플래그 — Top 5 중요 피처
> 동결 vs 신선 주기는 거의 모든 다른 컬럼의 해석을 바꿈

**모델 반영**: 원본 `해동된 배아 수`로 이미 학습. v6에선 명시 추가

#### ④ 남성요인 × ICSI 상호작용 (시술적합성)
```
남성요인 ✓ × ICSI ✓: AdequateICSI (적절)
남성요인 ✗ × ICSI ✓: OverusedICSI (과사용)
남성요인 ✓ × ICSI ✗: UnderservedICSI (필요한데 안 함)
남성요인 ✗ × ICSI ✗: Neither
```
**모델 반영 (v6)**: `시술적합성` 4범주 피처

## 5. v6 가이드 피처 추가 — 가이드 29개 전부 + 5개만 채택한 이유

### 5.1. 가이드 29개 파생변수 전체 목록

#### Group A. 시술·임신·출산 횟수 — 8개

| # | 변수명 | 한국어 의미 | 정의 | 도메인 의미 |
|---|---|---|---|---|
| 1 | `ever_delivered` | 과거 출산 경험 여부 | `총 출산 횟수 > 0` | **이 데이터셋 가장 강력한 단일 예측 변수** (Templeton 1996) |
| 2 | `repeated_3plus` | 3회 이상 반복 시술 여부 | `총 시술 횟수 ≥ 3` | 반복 실패군 (강한 부정) |
| 3 | `is_first_attempt` | 초회 시도 여부 | `총 시술 횟수 == 0` | 초회 시도 (가이드 +5.27%p) |
| 4 | `prev_birth_rate` | 이전 출산율 (스무딩) | `총 출산 / (총 임신 + 1)` | 임신 → 출산 전환율 |
| 5 | `유산_횟수` | 유산 횟수 | `총 임신 - 총 출산` | 반복 유산 시사 |
| 6 | `long_infertility` | 장기 난임 여부 | `임신 시도 연수 ≥ 7년` | Templeton 1996 임상 기준 |
| 7 | `censored_시술` | 시술 6회 이상 명시 플래그 | `총 시술 횟수 == 6` | '6회 이상' 검열 |
| 8 | `censored_임신` | 임신 6회 이상 명시 플래그 | `총 임신 횟수 == 6` | 동일 |

#### Group B. 배아 생성과 활용 — 9개

| # | 변수명 | 한국어 의미 | 정의 | 도메인 의미 |
|---|---|---|---|---|
| 9 | `is_FET` | 동결 배아 이식 여부 | `해동된 배아 수 > 0` | **★ Top 5 피처** — 동결 vs 신선 주기 |
| 10 | `성숙률` | 난자 성숙률 | `미세주입 난자 / (수집 신선 난자 + 1)` | 난자 성숙도 |

### 5.2. v3에서 5개만 채택한 이유

v3 노트북 작성자는 **가이드 29개 중 5개만 채택**. 이유는 **임상적 우선순위 + 트리 모델의 자동 학습 특성**을 고려:

| 채택 5개 | 채택 이유 | 보류한 24개 vs 비교 |
|---|---|---|
| `real_age_num` (실제 난자 제공자 나이) | **임신 성공의 가장 강한 예측 변수** — 자가/기증 구분 안 하면 큰 신호 손실 | 다른 피처는 부차적 |
| `days_mix_to_transfer` (배아 배양 기간) | **임상 결정 변수** — 3일(분할기) vs 5일(배반포)은 의사 판단 | 단순 시간 차이지만 임상적 의미 큼 |
| `log_n_embryo` (배아 수의 로그 변환) | **비선형 변환** — 배아 수의 한계효용 체감 (5→6 < 1→2) | 트리는 비선형 변환 못 함 → 명시 가치 |
| `miss_gene_test` (유전검사 결측 통합) | **결측 자체가 정보** — PGS/PGD/착상검사 결측 패턴이 클리닉 정책 시그널 | 다른 결측 플래그도 가능했으나 가장 강한 신호만 |
| `age_x_embryo_ratio` (나이 × 이식률) | **나이 × 이식률 상호작용** — 고령일수록 이식 개수 늘림 | 다른 상호작용은 트리가 자동 학습 가능 |

**보류한 24개의 공통 이유**: 가이드 Part 07이 명시:
> "최대 성능 추구 (XGBoost/LightGBM): 원본만 써도 충분 (D ≈ A, **트리는 상호작용 자동 학습**)"
>
> 가이드 실험: Model A (원본 3개) AUC 0.58600 vs Model D (원본 + 파생 5개) AUC 0.58613 → **차이 +0.013%p (거의 0)**

따라서 v3는 **트리가 자동 학습 못 하는 5개**(비선형 변환, 결측 패턴, 자가/기증 분기)만 추가했고, **자동 학습 가능한 단순 비율/조합 24개는 보류**.

### 5.3. v6에서 29개 모두 추가하니 어떻게 됐나

## 6. v8 Target Encoding 시도와 LB 결과

| 변수명 | 한국어 의미 | 대상 | smoothing |
|---|---|---|---|
| `TE__proc_specific` | 특정 시술 유형 평균 성공률 | `특정 시술 유형` (24범주) | 30 |
| `TE__proc_age` | 시술유형×나이 평균 성공률 | `시술 유형 × 시술 당시 나이` | 25 |
| `TE__proc_timing` | 시술유형×시기 평균 성공률 | `시술 유형 × 시술 시기 코드` | 25 |

**누수 방지 OOF 방식**:
```python
# 각 fold의 train fold에서만 평균 계산 → mapping
# va_idx에 매핑 (다른 fold의 정답 안 봄)
# test에는 fold별 mapping의 평균 사용
# smoothing으로 희귀 카테고리 안정화
```

## 7. v9의 돌파구 — CatBoost를 선택한 이유

### 왜 CatBoost를 시도했나

LightGBM과 XGBoost가 같은 패밀리라 앙상블 효과 약함을 v7 (3-way 앙상블, 미제출) 분석에서 확인. **진짜 다른 알고리즘이 필요**.

**CatBoost가 다른 이유**:

| 알고리즘 | 트리 구조 | 분기 결정 | Categorical 처리 |
|---|---|---|---|
| LightGBM | Leaf-wise (비대칭) | Histogram | 정수 매핑 |
| XGBoost | Level-wise (대칭) | Histogram | 원-핫 또는 정수 |
| **CatBoost** | **Oblivious tree (완전 대칭, 같은 split 반복)** | **Ordered statistics** | **Ordered Target Statistics** |

**Oblivious tree**: 같은 깊이 모든 노드가 같은 split feature/threshold 사용. **약한 학습기지만 일반화 강함** → 다른 종류의 실수 패턴.

**Ordered TS**: 범주형 target statistics 계산 시 **이전 행들의 정보만** 사용 → target leakage 자동 방지.

### 메모리 안전 설정

```python
CatBoostClassifier(
    iterations=600,
    depth=6,                  # LGBM의 num_leaves=95 대비 보수적
    border_count=64,          # 256 → 64 (양자화 줄임 → 메모리 ↓)
    max_ctr_complexity=0,     # CTR 비활성화 (대용량 메모리 ↓)
    bootstrap_type='Bernoulli',
    subsample=0.7,
    rsm=0.7,
    early_stopping_rounds=60,
)
```

메모리 사용량 ~900MB로 안정.

### 결과 — 단독은 평범, 앙상블에서 폭발

**CatBoost 단독 OOF**: 0.74045 (v3와 정확히 동일!)

**그러나 다른 모델과의 OOF 상관**:

| 모델 쌍 | 상관 |
|---|---|
| CatBoost ↔ LGBM v3 | **0.9928** |
| CatBoost ↔ LGBM v8 | **0.9925** |
| CatBoost ↔ XGB (v3 features) | **0.9932** |
| CatBoost ↔ XGB (v6 features) | **0.9931** |
| CatBoost ↔ 분리 모델 | **0.9916** |

이전 모델들끼리 상관 0.99~0.998이었는데, **CatBoost는 모두 0.991~0.993** — 확연히 낮음. **진짜 다른 신호를 잡고 있다는 증거**.

```python
from scipy.optimize import minimize

def neg_auc(w):
    w = np.clip(w, 0, None) / max(w.sum(), 1e-9)
    return -roc_auc_score(y_train, oof_array @ w)

# 5개 다른 시작점에서 Nelder-Mead 최적화
# 모든 시작점이 OOF 0.74100 근처로 수렴 → robust optimum 확인
```

## 10. Data Leakage 방지

대회 규정: "label encoding, get_dummies, scaling, 결측 처리에 test 사용 금지"

### 10.1. 점검 매트릭스

| 항목 | 위반 위험 | 우리의 방어 |
|---|---|---|
| 범주형 매핑 | `pd.get_dummies(pd.concat([train,test]))`는 위반 | train의 unique 값만 cat_maps 저장. test는 transform만. unseen → NaN |
| 결측 대치 | `train+test 평균으로 fillna`는 위반 | 결측 대치 안 함. 트리 native NaN 처리 |
| Scaling | `StandardScaler.fit(train+test)`는 위반 | 트리 모델은 scaling 불필요 |
| Target Encoding | `train 전체로 mapping` 후 test 적용은 위반 | Fold 내 train fold에서만. test는 fold별 평균 |
| CV 분할 | test를 CV에 포함 | StratifiedKFold(train만) |
| 앙상블 가중치 | test로 가중치 튜닝 | OOF에서만 scipy.optimize |

### 10.2. 코드 예시 — 안전한 범주형 처리

```python
# ✅ 안전 - train에서만 카테고리 수집
cat_maps = {}
for c in cat_cols:
    train[c] = train[c].astype(str).replace({'nan': np.nan})
    vals = sorted(train[c].dropna().unique())  # train 내부에서만
    cat_maps[c] = vals
    train[c] = pd.Categorical(train[c], categories=vals)

# ✅ 안전 - test에는 train의 카테고리만 적용
for c in cat_cols:
    test[c] = test[c].astype(str).replace({'nan': np.nan})
    vals = cat_maps[c]  # train의 카테고리 재사용
    unseen = (~test[c].isin(vals)) & test[c].notna()
    if unseen.any():
        test.loc[unseen, c] = np.nan  # 새 값은 결측으로
    test[c] = pd.Categorical(test[c], categories=vals)
```

```python
# ❌ 위반 예 - 절대 하지 않음
combined = pd.concat([train, test])
combined = pd.get_dummies(combined)  # test 정보가 train에 영향
```

### 10.3. 자동 패턴 스캔 결과

`grep` 패턴 검색 — **모든 학습 코드에서 0건 위반**:
- `pd.concat\(\[train.*test\]` ✅ 0건
- `get_dummies.*pd\.concat` ✅ 0건
- `LabelEncoder\(\)\.fit\(.*test` ✅ 0건
- `MinMaxScaler\(\)\.fit\(.*test` ✅ 0건

### 10.4. Target Encoding의 누수 검증

스모크 테스트:
- 같은 카테고리 행들이 다른 OOF TE 값을 가져야 함 (fold마다 train이 다르므로)
- 측정: **같은 카테고리 내 OOF TE std = 0.00841** (> 0)
- → 누수 없음 확인 (만약 누수면 std = 0)

## 11. 의사결정에서 배운 것들

### 11.1. 어떤 직관이 옳았나

| 직관 | 결과 | 교훈 |
|---|---|---|
| "v3는 이미 좋다, 안전망으로 유지" | ✅ | 검증된 베이스 보호가 핵심 |
| "도메인 가이드 다 넣으면 좋을 것" | ❌ | 가이드 본인 명시 잘 읽기 (Part 07) |
| "XGBoost 추가하면 다양성" | ⚪ | 같은 패밀리는 효과 약함 |
| "CatBoost는 진짜 다른 알고리즘" | ✅ | 알고리즘 패밀리 단위로 다양성 추구 |

### 11.2. 핵심 통찰 3가지

#### ① OOF 개선 ≠ LB 개선

**다양성이 진짜 추가될 때만 LB가 안정적으로 움직임**.

전이율:
- v3 → v8: OOF +0.00002, LB +0.00009 → 운 좋게 매칭
- v3 → v9: OOF +0.00055, LB +0.00019 → **35% 전이, 의미 있는 개선**

#### ② 앙상블 다양성은 알고리즘 패밀리 단위

LightGBM ↔ XGBoost 상관 0.9964 (같은 패밀리). CatBoost는 0.9916~0.9932 (다른 패밀리).

**같은 알고리즘 패밀리 안에서 모델 늘리는 건 한계**. 진짜 다양성은 **다른 트리 알고리즘** 또는 **다른 모델 클래스**.

#### ③ 트리 모델에 도메인 피처는 거의 무의미

가이드 Part 07 명시:
> "최대 성능 추구 (XGBoost/LightGBM): 원본만 써도 충분 (D ≈ A, 트리는 상호작용 자동 학습)"

v6에서 직접 검증 후 그대로였음.

**예외**: 트리가 자동 학습 못 하는 것들만 수동 추가 가치 있음:
- 비선형 변환 (`log_n_embryo`)
- 결측 패턴 (`miss_gene_test`)
- 조건부 분기 (`real_age_num` — 자가/기증)
